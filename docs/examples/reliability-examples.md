# Reliability — Before / After Examples

10 scenarios showing how machine-enforced rules and schema validation prevent failures.

---

| #   | Scenario                                                                        | ❌ Before (without FSH)                                                                                    | ✅ After (with FSH)                                                                                                                   |
| --- | ------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | **A junior dev imports `InvoiceAdapter` inside a Client Component**             | Silently works in dev; crashes in production when the browser tries to execute a Node.js module            | `eslint-plugin-boundaries` fails the lint check immediately — the PR cannot even be opened                                            |
| 2   | **The 101Digital API changes a field name from `dueDate` to `due_date`**        | The app silently receives `undefined` for due date; no error, just broken UI in production                 | `InvoiceResponseSchema` (Zod) throws a parse error at the adapter boundary — caught before reaching the UI                            |
| 3   | **A Route Handler returns a raw 101Digital error to the browser**               | The browser's Network tab exposes internal error messages, stack traces, and upstream service names        | The BFF adapter maps errors to safe responses; internal details never leave the server                                                |
| 4   | **A dev stores `access_token` in `localStorage` for convenience**               | Token is readable by any JavaScript on the page — one XSS attack steals all sessions                       | There is no `localStorage` code path in the architecture; the only storage is `HttpOnly` cookies set by the server                    |
| 5   | **A feature imports state from another feature**                                | A bug fix in `features/auth` silently breaks `features/create-invoice` — discovered only in QA             | `eslint-plugin-boundaries` rejects the import at lint time with a clear "cross-feature import forbidden" message                      |
| 6   | **A required env var (`DIGITAL_CLIENT_SECRET`) is missing in a new deployment** | The app boots, the first login attempt throws an unhandled exception in production                         | `env.server.ts` validates all required vars at startup with Zod — the process exits with a clear error before accepting traffic       |
| 7   | **A user submits a form with a negative invoice total**                         | Client-side validation is bypassed via devtools; the raw value reaches 101Digital                          | The Route Handler re-validates with the same Zod schema; negative totals are rejected server-side regardless of what the client sends |
| 8   | **A new developer adds `NEXT_PUBLIC_DIGITAL_CLIENT_SECRET` to expose a key**    | The secret is bundled into the client JavaScript and visible to anyone who opens DevTools → Sources        | The naming convention `NEXT_PUBLIC_` is documented as forbidden for secrets; `env.server.ts` only ever reads non-public vars          |
| 9   | **Two developers change the same API contract independently**                   | The frontend sends `customerName`, the Route Handler expects `customer_name` — mismatch only caught in E2E | `packages/api-contracts` is the single source of truth; both sides import the same Zod schema — TypeScript errors if they diverge     |
| 10  | **A race condition leaves the invoice list in a stale state**                   | Unclear which hook owns the refetch logic — multiple places trigger updates inconsistently                 | TanStack Query is the single cache owner; `useInvoiceList` is the only place list state lives — invalidation is deterministic         |
