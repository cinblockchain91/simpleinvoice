# Security by Design — Before / After Examples

10 scenarios showing how the architecture prevents vulnerabilities structurally, not by policy.

---

| #   | Scenario                                                                                          | ❌ Before (without FSH)                                                                                                       | ✅ After (with FSH)                                                                                                                                                                   |
| --- | ------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | **An XSS payload runs in the browser and tries to steal the session token**                       | Token is in `localStorage` — `document.cookie` or `localStorage.getItem("token")` exposes it instantly                        | Token lives in an `HttpOnly` cookie — no JavaScript API can read it; the attack yields nothing                                                                                        |
| 2   | **A developer adds `NEXT_PUBLIC_DIGITAL_CLIENT_SECRET` to expose the OAuth secret**               | The secret is bundled into the client JavaScript and visible to any user who opens DevTools → Sources                         | Naming convention and `env.server.ts` enforce server-only access; a `NEXT_PUBLIC_` secret would be caught in code review and is architecturally discouraged                           |
| 3   | **A client component calls the 101Digital API directly to bypass the BFF**                        | `fetch("https://api.101digital.io/invoice-service/...")` works from the browser — tokens and secrets must be exposed to do it | `eslint-plugin-boundaries` prevents client code from importing `infrastructure/`; there is no client-side path to 101Digital — only `/api/*` routes exist                             |
| 4   | **A user manipulates the form payload via DevTools and sends a negative invoice total**           | Client-side Zod validation is bypassed; the manipulated payload reaches 101Digital                                            | The Route Handler re-runs the same Zod schema from `packages/api-contracts` — invalid payloads are rejected at the server boundary, regardless of what the client sends               |
| 5   | **A new Route Handler is added without authentication**                                           | No middleware enforces auth; the unprotected handler leaks invoice data to unauthenticated users                              | Auth middleware checks `access_token` cookie before every request to `/(dashboard)` routes; new handlers inside that group are protected automatically                                |
| 6   | **A 101Digital API error response containing internal service names is forwarded to the browser** | `res.json(upstreamError)` exposes internal infrastructure details — useful for attackers mapping the system                   | `InvoiceAdapter.ts` catches upstream errors and maps them to safe `Result.fail(InvoiceError.NotFound)` — no raw upstream response ever reaches the client                             |
| 7   | **The `org_token` is accidentally logged in a server-side console statement**                     | Token appears in Vercel logs, accessible to anyone with log access                                                            | The token only exists in `SessionCookieStore` during a single request lifetime; it is never assigned to a named variable in application code — hard to accidentally log               |
| 8   | **A developer copies a working fetch snippet that puts the token in the URL query string**        | `fetch("/api/invoices?token=" + token)` — token appears in server logs, browser history, and Referer headers                  | The architecture has no mechanism to retrieve the token from client code; `bff-client.ts` sends cookies automatically via `credentials: "include"` — no manual token handling needed  |
| 9   | **A supply-chain attack injects malicious code into a UI component library**                      | The compromised library has access to `localStorage`, cookies (if not HttpOnly), and can make authenticated requests          | Auth tokens are `HttpOnly` — the compromised library cannot read them; all requests require the cookie to be sent automatically by the browser, which it does only to the same origin |
| 10  | **A penetration tester attempts to access `/api/invoices` without an `access_token` cookie**      | Without a strict auth check, the handler may return data or expose a server error                                             | The Route Handler reads from `SessionCookieStore`; if `access_token` is absent, it returns `401` before any adapter or domain code is reached                                         |
