# Maintainability — Before / After Examples

10 scenarios showing how clear ownership and layer boundaries reduce the cost of change.

---

| #   | Scenario                                                                  | ❌ Before (without FSH)                                                                                                                   | ✅ After (with FSH)                                                                                                                     |
| --- | ------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | **A new developer asks "where does invoice validation live?"**            | They grep the codebase and find validation scattered across a form component, a route handler, and a custom hook — all slightly different | One answer: `packages/domain/src/invoice/` — every business rule for invoices lives here and nowhere else                               |
| 2   | **The team decides to remove the "search invoices" feature**              | Must grep for all usages, find tangled imports across components, hooks, and API calls, then carefully unpick them                        | Delete `features/search-invoices/`. ESLint confirms no other feature imports it. Done                                                   |
| 3   | **101Digital changes its base API URL**                                   | The URL is hardcoded (or `.env`-referenced) in multiple hooks, components, and fetch calls across the app                                 | Change one line in `env.server.ts` — the adapters read it; no feature code knows the URL exists                                         |
| 4   | **Onboarding a new team member**                                          | They ask "where do I put this new component?" — the answer depends on tribal knowledge and inconsistent convention                        | The FSH layer diagram in `docs/02-architecture.md` answers every "where does this go?" question unambiguously                           |
| 5   | **A bug is found in how invoice tax is calculated**                       | Bug could be in the UI component, the hook, the route handler, or the API adapter — takes time to locate                                  | Tax calculation is in `packages/domain/src/invoice/Invoice.ts` — one file, one fix, covered by one unit test                            |
| 6   | **The design team replaces shadcn/ui with a different component library** | UI library classes and component APIs are mixed into business logic hooks — replacing them touches dozens of files                        | Swap `shadcn/ui` components in `shared/ui/` and `shadcn/`; domain, feature models, and entities are untouched                           |
| 7   | **The API renames `customerName` to `customer_name` in responses**        | Multiple components and hooks reference the field directly — each must be found and updated                                               | Only `InvoiceAdapter.ts` maps the external field to the domain model — one change propagates everywhere                                 |
| 8   | **A mandatory new field `purchaseOrderNumber` must be added to invoices** | Must update: the form component, the hook, the API call, the type, the validation, the list table, the detail view — easy to miss one     | Add to `Invoice.ts` (entity) → TypeScript immediately flags every consumer that doesn't handle it — the compiler creates the to-do list |
| 9   | **A code review for a "simple" feature change**                           | The PR touches hooks, components, API calls, and types in mixed directories — reviewer must understand the full app to assess impact      | The PR touches only files within `features/create-invoice/` — the reviewer knows the blast radius is bounded by the layer rule          |
| 10  | **Six months later, understanding why a business rule exists**            | The rule is embedded in a component render function with no context — git blame leads to an ambiguous commit message                      | The rule is in `packages/domain` with a named function; its test file documents the exact edge case it was written to handle            |
