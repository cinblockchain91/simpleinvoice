# Framework Independence — Before / After Examples

10 scenarios showing how zero-dependency domain logic survives platform changes.

---

| #   | Scenario                                                                           | ❌ Before (without FSH)                                                                                                              | ✅ After (with FSH)                                                                                                                        |
| --- | ---------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------ |
| 1   | **Next.js 17 ships breaking changes to App Router**                                | Business logic is in Server Components, Route Handlers, and hooks mixed with Next.js APIs — entire app must be audited and rewritten | Only `apps/web` adapter layer changes; `packages/domain` is untouched — migration is bounded and testable                                  |
| 2   | **The team evaluates switching from Next.js to Remix**                             | Fetching, routing, and business logic are intertwined — a proof-of-concept migration rewrites 80% of the codebase                    | Build a `apps/remix` that imports `packages/domain`; domain logic works on day one — evaluate only the framework-specific layer            |
| 3   | **React 20 deprecates a core API used throughout the app**                         | Hooks, context patterns, and component APIs are scattered — impact analysis takes days                                               | `packages/domain` has zero React imports; only presentation layer components need updating                                                 |
| 4   | **The company wants to run business logic on Deno for a serverless edge function** | Logic is wrapped in `getServerSideProps`, Route Handlers, or React hooks — impossible to extract and run on Deno                     | `packages/domain` is valid TypeScript with no Node.js built-ins — runs on Deno `import` without modification                               |
| 5   | **A security audit requires moving from Next.js to a hardened custom server**      | The server framework and the application logic are one and the same — impossible to separate cleanly                                 | The hexagonal boundary already separates them; swap the delivery mechanism while keeping the domain intact                                 |
| 6   | **The team wants to test the domain layer on Bun runtime**                         | Tests depend on Jest's JSDOM, React Testing Library, and Next.js mocks — cannot run outside Node.js/Jest                             | `packages/domain` tests use only `vitest` and plain TypeScript — run `bun test` and they pass unchanged                                    |
| 7   | **Five years later, React is no longer the dominant UI framework**                 | The entire codebase is tied to React idioms — migrating to a new framework is a full rewrite                                         | `packages/domain` is framework-agnostic TypeScript; write a new presentation layer in whatever is current while keeping all business logic |
| 8   | **A new developer joins from a Vue.js background**                                 | They must understand Next.js-specific patterns just to fix a bug in invoice date validation                                          | They edit `packages/domain` — pure TypeScript classes and functions, no framework knowledge required                                       |
| 9   | **The CI pipeline needs to run domain tests 10× faster**                           | Tests spin up a full Next.js environment with JSDOM — cannot be parallelized without major configuration                             | Domain tests are plain `vitest` with no DOM — run in worker threads by default, no special config needed                                   |
| 10  | **The app must run offline as a PWA with local data processing**                   | Fetching and processing logic is tied to server-side Route Handlers and SSR — difficult to adapt for offline use                     | `packages/domain` validation and calculation functions run in the browser service worker with no modification                              |
