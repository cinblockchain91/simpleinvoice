# Scalability — Before / After Examples

10 scenarios showing how the monorepo structure and strict boundaries support team and platform growth.

---

| #   | Scenario                                                                               | ❌ Before (without FSH)                                                                                                             | ✅ After (with FSH)                                                                                                                                        |
| --- | -------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | **The company decides to ship a React Native mobile app**                              | Must rewrite all business logic (validation, calculations, use cases) for the mobile codebase — two codebases diverge immediately   | `apps/mobile` installs `packages/domain` — all validation, use cases, and types are already there, zero rewrite                                            |
| 2   | **Team A and Team B are building features in parallel**                                | Both teams edit shared hooks and components; merge conflicts appear weekly; one team's refactor silently breaks the other's feature | Team A owns `features/create-invoice/`, Team B owns `features/list-invoices/` — cross-feature imports are forbidden, conflicts are structurally impossible |
| 3   | **The same invoice date validation must run on web, mobile, and a future admin panel** | Each platform implements its own version — bugs are fixed in one place and forgotten in two others                                  | `packages/domain/src/invoice/` holds the single implementation; all three platforms import it — one fix reaches everywhere                                 |
| 4   | **The company wants a Tauri desktop app**                                              | Must re-implement the BFF pattern and business rules for Electron/Tauri from scratch                                                | `apps/desktop` wraps `apps/web` build; shares `packages/domain` — no business logic is re-implemented                                                      |
| 5   | **The engineering team grows from 2 to 10 developers**                                 | No clear ownership model; PRs step on each other; "who owns this file?" becomes a daily question                                    | Each FSD feature slice has an unambiguous owner; the dependency rule prevents ownership from blurring over time                                            |
| 6   | **Adding a completely new feature ("recurring invoices")**                             | Must figure out where the new state, API calls, UI, and types go — risky to do it wrong and hard to fix later                       | Create `features/recurring-invoices/` with `ui/`, `model/`, `index.ts` — the structure is defined by the architecture                                      |
| 7   | **Incremental build times grow as the codebase scales**                                | A change to a shared utility triggers a full rebuild of all features                                                                | Turborepo tracks the dependency graph between packages — changing `packages/domain` only rebuilds packages that depend on it                               |
| 8   | **A backend team wants to consume the same domain types in a Node.js microservice**    | Domain types are tied to React component props and Next.js types — unusable outside the app                                         | `packages/domain` is pure TypeScript with no framework deps — the Node.js service installs it from the monorepo directly                                   |
| 9   | **The product adds a CLI tool for batch invoice generation**                           | Business logic is inseparable from React hooks and components — cannot run in a terminal                                            | `packages/domain` runs on plain Node.js; the CLI imports `CreateInvoiceUseCase` directly                                                                   |
| 10  | **Performance profiling reveals the invoice list is slow**                             | Hard to isolate — is it the fetch logic, the sorting, the component re-renders, or the pagination?                                  | Each concern lives in its own layer: `useInvoiceList` (fetch), page component (sort/render), `InvoicePagination` (UI) — profile and optimize independently |
